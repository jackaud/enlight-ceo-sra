# ADR-004: 前端聊天 + 表单交互模式

- **状态**: 已采纳
- **日期**: 2026-03-14
- **决策者**: Jack
- **关联项目**: EnlightIn（领导力评估咨询管理软件）
- **关联决策**: ADR-001（表单定义与管理）、ADR-003（Agent 架构模式）

## 背景

EnlightIn 的核心交互是 Agent 通过对话引导用户完成领导力评估。评估过程中，Agent 需要在对话流中嵌入结构化表单（量表评分、开放式问题）。需要确定：

1. 表单在哪里呈现（内嵌对话 vs 弹窗 vs 侧边栏）
2. 表单显示时聊天是否锁定
3. 连续多个表单的交互节奏
4. 表单提交/取消后对话如何继续
5. 如何展示评估进度

### 参考实现：Carevol 的交互模式

Carevol 的表单交互已经过生产验证，核心模式：

- 表单**内嵌在对话流中**，紧跟触发它的 Assistant 消息下方
- 表单在消息**流式输出完成后**才出现（`AI_CHAT_COMPLETE` 事件携带）
- 聊天输入**不被锁定**，表单显示时用户仍可打字
- 提交后表单**原位替换为成功卡片**
- 取消后表单**消失**，Agent 收到通知继续对话
- 通过 `messageId` 关联表单与消息，一条消息可挂多个表单

## 决策

### 1. 表单呈现方式：内嵌对话流

**采用 Carevol 的内嵌模式，不使用弹窗或侧边栏。**

```
┌──────────────────────────────────────┐
│ 🤖 接下来评估"沟通能力"维度。       │  ← Assistant 消息
│    请根据日常观察进行评价：          │
├──────────────────────────────────────┤
│ ┌──────────────────────────────────┐ │
│ │ 📋 领导力评估 - 沟通能力        │ │  ← 表单，内嵌在消息下方
│ │                                  │ │
│ │ 该领导能清晰传达团队目标:        │ │
│ │ ○1  ○2  ○3  ○4  ●5             │ │
│ │                                  │ │
│ │ 请举一个具体的例子:              │ │
│ │ [                              ] │ │
│ │                                  │ │
│ │ [提交评价]  [跳过此维度]         │ │
│ └──────────────────────────────────┘ │
├──────────────────────────────────────┤
│ 💬 输入消息...                [发送] │  ← 输入框不禁用
└──────────────────────────────────────┘
```

**理由**：
- 评估本质上是一场"对话式访谈"，表单是对话的一部分，不应打断对话流
- 弹窗会遮挡上下文，用户看不到 Agent 之前说了什么
- 侧边栏在移动端体验差
- Carevol 已验证此模式可行

### 2. 聊天输入行为：不锁定

**表单显示时聊天输入保持可用。**

用户可能在填表单时想说"等一下，我先确认个事情"或"这个维度我不太确定怎么评"。锁定输入会阻断这种自然交流。

### 3. 连续表单的交互节奏：Agent 自主决定

EnlightIn 的 collecting 阶段会连续触发多个维度的评估表单，交互节奏如下：

```
🤖 我们先评估"沟通能力"维度。
📋 [沟通能力量表]                      ← 表单

（用户提交，表单变成成功卡片 ✅）

🤖 谢谢。你给沟通能力打了 2 分，              ← Agent 根据评分决定是否追问
   能具体说说哪些方面让你觉得不够清晰吗？

👤 上次项目复盘会他讲了20分钟没人听懂重点。

🤖 明白了，这个反馈很有价值。                  ← Agent 消化后进入下一维度
   接下来评估"愿景规划"能力。
📋 [愿景规划量表]                      ← 下一个表单

（用户提交，成功卡片 ✅）

🤖 好的。接下来是"决策能力"。                  ← 评分正常，不追问，直接下一个
📋 [决策能力量表]
```

**关键设计**：

- 每次只展示一个评估表单，不同时展示多个
- 表单提交后，**Agent 自主决定**是追问还是直接进入下一维度
  - 极端评分（1 分或 5 分）→ LLM 倾向追问"能举个例子吗？"
  - 中间评分（3 分）→ 可以直接进入下一维度
  - 这个判断由 LLM 在 prompt 引导下自然完成，不需要硬编码
- 这种"表单 → 对话 → 表单"的交替节奏让评估感觉像访谈，而非填问卷

### 4. 表单提交后的反馈

**沿用 Carevol 的成功卡片模式，原位替换表单。**

```
提交前:
┌──────────────────────────────────┐
│ 📋 领导力评估 - 沟通能力        │
│ [表单字段...]                    │
│ [提交评价]  [跳过此维度]         │
└──────────────────────────────────┘

提交后（原位替换）:
┌──────────────────────────────────┐
│ ✅ 沟通能力评估已完成             │
│ 评分: 4/5                        │
│ 已记录开放式反馈                 │
└──────────────────────────────────┘
```

### 5. 表单取消语义：跳过维度

**Carevol 的"取消"= 放弃操作。EnlightIn 重新定义为"跳过此维度"。**

| 操作 | 行为 | Agent 响应 |
|------|------|-----------|
| 提交 | 保存评分，显示成功卡片 | Agent 决定追问或进入下一维度 |
| 跳过此维度 | 表单消失，不记录数据 | Agent 确认"好的，跳过沟通能力"并进入下一维度 |
| 用户发消息"暂停评估" | 不触发表单事件 | Agent 通过对话理解意图，暂停流程 |

按钮文案从"取消"改为**"跳过此维度"**，语义更明确。

如果用户想暂停整个评估（而非跳过单个维度），通过对话表达即可，Agent 会理解。评估状态通过 LangGraph Checkpoint 持久化，用户随时可以回来继续。

### 6. 评估进度展示

**在对话区域上方固定显示进度条。**

```
┌──────────────────────────────────────┐
│ 📊 评估李明 - 沟通 ✅ 愿景 ✅ 决策 🔵 团队 ○ 情商 ○ 担当 ○  │
├──────────────────────────────────────┤
│                                      │
│ （对话流）                           │
│                                      │
└──────────────────────────────────────┘
```

进度数据来源：
- StateGraph 的 `state.current_dimension` 和 `state.dimensions`（参见 ADR-003）
- 通过独立的 WebSocket 事件推送（如 `assessment_progress`），不复用表单事件通道
- 前端用固定位置的进度组件渲染，不在对话流内

### 7. 事件时序

```
用户发送消息 / 提交表单
  ↓
后端 ai_chat_start           → 前端显示打字指示器
  ↓
后端 ai_chat_chunk × N       → 前端逐字渲染 Agent 回复
  ↓
后端 ai_chat_complete         → 前端：
  (携带 ui_forms 数组)           1. 完成消息渲染
  (携带 assessment_progress)     2. 在消息下方渲染表单（如果有）
                                 3. 更新进度条
                                 4. 隐藏打字指示器
  ↓
用户填写表单 → 点击"提交评价"
  ↓
前端：
  1. 表单进入 loading 状态
  2. 发 WebSocket ai_ui_form_submit 事件
  3. 表单变为成功卡片
  ↓
后端收到提交：
  1. 评估数据存入 assessment_responses 表
  2. 更新 StateGraph state（collected_responses）
  3. route_after_collecting 判断是否进入下一阶段
  4. Agent 继续对话（追问或下一维度）
```

### 8. 前端状态管理

借鉴 Carevol 的三层 Hook 架构：

| Hook | 职责 |
|------|------|
| `useFormState` | 管理表单生命周期（`Map<messageId, FormState[]>`），处理提交/取消/loading 状态，发 WebSocket 事件 |
| `useFormRenderer` | 提供 `renderFormForMessage(messageId)` 回调，解析 form_id 找组件（预定义或动态渲染器），渲染成功卡片 |
| `useAssessmentProgress` | EnlightIn 新增，管理评估进度状态，监听 `assessment_progress` 事件 |

## 与其他 ADR 的关系

- **ADR-001**：定义了表单的数据结构（UIFormSchema、UIFormResult）和预定义+动态混合模式。本 ADR 定义表单在前端的**呈现和交互方式**。
- **ADR-003**：定义了 StateGraph 的阶段划分。本 ADR 中的进度展示依赖 StateGraph 的 `phase` 和 `current_dimension` 状态。
